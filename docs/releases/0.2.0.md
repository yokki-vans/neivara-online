# 0.2.0 — equipment foundation

Historical release note: the five-class/five-people content catalog described below was replaced by the five-race, two-gender, two-class matrix in 0.3.0. This file records the state of 0.2.0 and is not current gameplay documentation.

Status: release candidate for the public technical alpha.

## Player-facing changes

- 34 original items across weapons, armor, accessories, consumables and materials;
- 13 equipment slots, searchable paper-doll inventory, comparison and derived stats;
- starter kits for all five classes, consumables and safe enhancement progression;
- equipped weapon and armor visibly change the procedural 3D character;
- distinct visual traits for all five peoples and dedicated weapon silhouettes;
- original ground/path textures and clean-room character/equipment concepts.

## Server and operations

- unique item instances, equipment, consumable cooldowns and economy ledger in PostgreSQL;
- serialized character mutations, atomic gold deltas and retry-safe high-value actions;
- authoritative equipment stats in PvE/PvP, range, hit, mitigation, haste and loot;
- weapon-specific basic range and attack interval, acknowledged by the server after validation;
- PostgreSQL CI integration test, migration lock, readiness check and CycloneDX SBOM;
- protocol version 2 rejects an incompatible cached client with a refresh message.

## Compatibility

The release applies append-only database migrations during server startup. A 0.1 client is intentionally incompatible with protocol 2 and must reload the GitHub Pages client. Runtime art paths use `import.meta.env.BASE_URL` and remain compatible with the repository subpath.

Do not roll the 0.2 server binary back to 0.1 after accepting writes: 0.2 makes `item_instances` canonical while 0.1 reads legacy `inventory_stacks`, so inventory would appear stale. Recover by fix-forward on 0.2+ or restore the pre-upgrade database snapshot together with the old binary.

## Known release gates not yet satisfied

- procedural heroes are not final rigged/animated GLB models;
- the return-stone cast is disabled until interruption-safe server casting exists;
- temporary elixir buffs are session effects: re-use refreshes duration and a two-minute reconnect grace preserves buffs/cooldowns/death state; a full server restart clears non-durable effects;
- party, friends, trade, auction, guilds and broad quest content remain alpha/beta work;
- backup restore drills and measured 30-player soak/load targets are still pending.

## Go/no-go checklist

Repository gates implemented for every candidate commit:

- [x] PostgreSQL integration migration/transaction test is part of `CI / verify`;
- [x] high/critical `npm audit` is blocking;
- [x] the final Docker runtime target is built in CI;
- [x] typecheck, lint, tests, asset provenance, production builds, SBOM and CodeQL are present;
- [x] third-party GitHub Actions are pinned to commit SHAs resolved from their release tags.

Manual production gates (release remains **no-go** while any item is unchecked):

- [ ] protect `main` and require the exact `CI / verify` and `CodeQL / analyze` check contexts;
- [ ] prove a failing `main` check blocks both GitHub Pages and Railway server deployment;
- [ ] set and verify Railway at exactly one server replica in one region with Wait for CI enabled;
- [ ] record candidate commit SHA, Docker digest and before/after migration versions;
- [ ] create a pre-0.2 Railway snapshot plus encrypted logical dump and verify their hashes/retention;
- [ ] restore that backup into an isolated database and complete auth/inventory/economy smoke tests within the four-hour RTO target;
- [ ] rehearse the 0.2 migration on restored data and approve fix-forward versus matched snapshot+0.1 rollback;
- [ ] complete the 30-player two-hour soak/reconnect/duplication test and attach measurements;
- [ ] replace procedural heroes with reviewed, licensed/provenanced production GLB assets;
- [ ] complete the post-deploy health/smoke checks and 15-minute integrity watch.

Operational commands, evidence fields and the 0.2 rollback boundary are defined in `docs/OPERATIONS.md`.

This candidate is a hardened playable foundation, not a claim that a content-complete MMORPG has been produced in one iteration.
